# Executive Summary
Sentinel AI is a security scanner purpose-built for AI-generated ("vibe-coded") applications. Developers now ship huge amounts of code they didn't fully write and don't fully understand — generated by Copilot, Cursor, Claude, or ChatGPT. This code tends to carry a specific, repeatable set of vulnerabilities: hardcoded secrets, insecure auth scaffolding, unvetted AI-suggested dependencies, and permissive defaults that "looked right" to the model.

Sentinel AI scans a GitHub repository, detects these vibe-coding-specific vulnerability patterns using a combination of static analysis and LLM-based reasoning, and produces a prioritized, actionable remediation report — in minutes, not a security audit cycle.

The Hackathon MVP focuses on doing **one thing convincingly**: scanning a real repo and catching real, verifiable vulnerabilities live in front of judges.

---

# Problem Statement
AI coding assistants have changed how software gets written, but security practices haven't caught up.

- Developers increasingly ship code they didn't write line-by-line and can't fully audit.
- LLMs reliably reproduce certain unsafe patterns: hardcoded API keys, weak auth checks, overly permissive CORS/config, insecure defaults copied from training data, and dependencies suggested without vetting.
- Traditional security tools (Snyk, SonarQube, generic SAST) weren't designed to look for *AI-authorship-specific* risk patterns — they treat all code the same.
- Fewer teams are building security tooling for this exact gap, even though the underlying problem (LLM-generated apps shipping to production) is growing fast.

Result: a rising volume of production applications with a predictable, semi-invisible class of vulnerabilities, and no tool specifically tuned to catch them.

---

# Proposed Solution
Sentinel AI imports a GitHub repository once and runs a focused two-layer scan:

1. **Static Detection Layer** — deterministic checks for hardcoded secrets, exposed environment variables, dangerous config flags, and known-risky dependency patterns (regex/entropy-based, not LLM-guessed).
2. **AI Reasoning Layer** — an LLM agent reviews flagged areas and surrounding code context to explain *why* something is risky, assess severity, and suggest a concrete fix — with attention to patterns common in AI-generated code specifically (e.g., placeholder secrets left in, auth checks that look complete but short-circuit, copy-pasted boilerplate with unsafe defaults).

Output: one **Prioritized Security Report** ranking findings by real exploitability, not just pattern-matching noise.

# MVP Scope

## Included

### Authentication
- GitHub OAuth login

### Dashboard
Users can:
- Import a repository
- Trigger a scan
- View live scan progress
- View past reports

### Repository Import
Supported:
- Public GitHub repositories

### Static Detection Engine
Detects:
- Hardcoded secrets/API keys (entropy + pattern-based, e.g. TruffleHog-style)
- `.env` files committed / missing from `.gitignore`
- Dangerous config flags (e.g. `DEBUG=True` in prod configs, permissive CORS)
- Known-risky dependency patterns (outdated/abandoned packages, typosquat-adjacent names)

### AI Reasoning Agent
For each static finding:
- Explains the risk in plain language
- Assesses real exploitability (not just "pattern matched")
- Suggests a concrete fix
- Flags additional context-based risks static rules can't catch (e.g. an auth check that exists but is logically bypassable)

### Prioritized Security Report
Sections:
- Executive Summary
- Risk Score (overall)
- Findings, ranked by severity
- Fix Recommendations
- Export (PDF/Markdown)

# Functional Requirements
- **FR-001** Users can authenticate via GitHub OAuth
- **FR-002** Users can import a public repository
- **FR-003** System runs static detection across the repo
- **FR-004** AI Reasoning Agent evaluates and explains each static finding
- **FR-005** System produces a ranked, prioritized report
- **FR-006** Users can export the report (PDF or Markdown)
- **FR-007** Scan progress is shown live in the UI

---

# Non-Functional Requirements
- Fast: scan of a typical repo completes in under 2 minutes
- Findings must be **real**, not fabricated — false positives should be labeled with confidence, not hidden
- Demo-reliable: works consistently against a pre-tested target repo
- Extensible: detection rules and agent prompts are modular for post-hackathon growth

# Technology Stack

## Frontend
- Next.js, TypeScript, Tailwind CSS

## Backend
- FastAPI, Python

## AI
- **GPT-5.6** — powers the AI Reasoning Layer (finding explanation, severity scoring) and the Remediation Engine (fix generation)
- **Codex** — used as the primary build agent throughout development (scaffolding services, writing detectors, generating the remediation diffs logic itself)

## Detection Tooling
- Regex/entropy-based secret detection (TruffleHog-style approach)
- Tree-sitter / Python AST for code parsing
- ripgrep for fast repo-wide pattern search

## Database
- PostgreSQL (findings, reports, projects)

## Auth
- GitHub OAuth

## Deployment
- Frontend: Vercel
- Backend: Railway
- Database: Neon PostgreSQL
